# Spec: Stage 2 - Outbound Calls (EPIC-002)

**Date:** 2026-06-11
**Status:** Draft (correct before implementation; recheck provider pricing/models first)
**Depends on:** Stage 1 control plane; D-5 (Pipecat), D-8 (Python)

## 1. Goal

Replace the stub worker with a real Pipecat voice worker: a queued task produces a real outbound call via Twilio with live transcript, AI disclosure, policy hooks, and a post-call summary.

## 2. Scope

In scope:

- Twilio integration: number purchase, outbound dialing via Programmable Voice/SIP, media streams into the worker.
- Pipecat pipeline: Twilio transport <-> Deepgram STT <-> LLM agent <-> Cartesia TTS.
- Mandatory AI disclosure as the first assistant utterance, in the call language (ES/EN/RU), non-overridable by prompt.
- Call state machine: dialing -> connected -> disclosure -> conversation -> (approval pause) -> wrap-up -> ended; plus failed/no-answer/voicemail branches.
- Transcript segments streamed to the control plane via the Stage-1 event contract (unchanged).
- Live-call web page (replaces Stage-1 stub): live transcript, status, Hang up; Approve/Reject inherited; Whisper instruction (text injected into agent context); Take over deferred to Stage 3.
- Post-call summary + next steps generated by LLM, stored on task_run, pushed to Telegram.
- Retry policy: busy/no-answer -> bounded retries with backoff; mid-call drop -> task_run preserves partial transcript and state (recoverability NFR).
- First scenario: restaurant booking (lowest risk, autonomy level 1).

Out of scope: inbound calls (004), full policy rules (003), hold-music/cost optimization (006), calendar write (005).

## 3. Voice Worker Design

- One process, asyncio; one active call per worker instance initially (scale by N processes later).
- Consumes Redis queue jobs; claims task_run; opens Twilio call; on terminal state pushes summary event.
- LLM agent: system prompt assembled from structured_goal + allowed profile_facts + policy preamble; tool calls: `request_approval`, `end_call`, `log_fact`, `propose_summary`.
- Every LLM tool call passes through `packages/policy.evaluate()` before execution.
- Language: ES default for targets in Spain; configurable per task.
- Latency budget: target <1s assistant response; measure STT/LLM/TTS stage timings per turn (stored for EPIC-006).

## 4. Telephony Details

- One Spanish Twilio number (~$1-2/mo). Caller ID = assistant's own number (not user's) for MVP.
- Recording: OFF by default until EPIC-007 resolves consent rules; transcript-only.
- Webhooks (status callbacks) go to FastAPI; media stream WebSocket goes directly to the voice worker (Cloudflare Tunnel in dev, VPS in prod-like).

## 5. Acceptance Criteria

1. Telegram task -> real call to a test phone; callee hears AI disclosure in the right language.
2. Live transcript appears on the web page within ~2s of speech.
3. Mid-call `request_approval` pauses the conversation gracefully ("let me check, one moment") and resumes after Telegram approve.
4. Hang up from web works; summary arrives in Telegram after the call.
5. Killing the worker mid-call leaves task_run `failed` with partial transcript intact.
6. A real restaurant booking completed end-to-end at least once.
